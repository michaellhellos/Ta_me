# Spec 030: React Memory Hardening against Token Bleeding

## 1. Prompt & Objective
**Prompt:** "{bantu saya yang di bagian mentor nya yang di bagian chat nya itu mengapa ya kalau saya ingin membalas chat siswa saya yang muncul di paling atas itu nama saya sendiri... dan juga misal mengapa ya saya login sebagai email nya bejo nah itu kan role nya mentor lalu saya login juga sebagai user biasa saya chat mentor nya..."
**Tujuan:** Membasmi anomali reaktif pada komponen *React* di mana *Chat Header* (Nama Lawan Bicara) dievaluasi memutar menjadi nama sang *Sender* (Diri Sendiri), serta fenomena transmisi pesan nyasar ke Inbox Mentor yang salah akibat `localStorage` *Collision*.

## 2. Analisa Akar Masalah (Root Cause)

Satu-satunya penyebab mengapa Mentor "Bejo" melihat pesan yang dikirim oleh "User Biasa" kepada "Michaell" adalah karena **Metode Testing Konkuren pada 1 Browser**.

**Mekanisme Kerusakan (Token Bleeding):**
1. User login sebagai "Bejo". Tab 1 membuka laman Mentor.
2. User membuka Tab 2 dan login sebagai "User Biasa".
3. Secara sistematis, Tab 2 membunuh `localStorage.token` milik Bejo dan menggantinya dengan Token milik User Biasa.
4. Ketika layar "Bejo" di Tab 1 terjadi *re-rendering* sekecil apapun (contoh: *loading state*, atau mengetik huruf di input), `Chat.tsx` dan `Qna.tsx` secara proaktif membaca ulang `localStorage`.
5. Tab 1 milik Bejo seketika menggunakan Token milik Tab 2. **Identitas Bejo hilang.** 
6. Setiap klik, `fetch`, atau evaluasi `myId` di Tab 1 kini dieksekusi dengan *mindset* bahwa Tab 1 adalah "User Biasa". Inilah alasan mengapa Inbox Bejo tiba-tiba memuat daftar chat milik User Biasa! Dan karena Tab 1 masih merasa antarmukanya adalah layar Mentor, sistem mengevaluasi Lawan Bicara secara terbalik!

## 3. Rencana Perubahan Code (The Cure)

Meskipun kesalahan ini 100% dipicu oleh tata cara uji coba tab-ganda, saya akan menambah "Armor" kebal-mutasi pada komponen UI Anda.

Metode yang digunakan adalah **React `useMemo` Caching Protocol**:
*   `Frontend/frontend/src/pages/Chat.tsx`: Membungkus deklarasi `token`, `user`, dan `myId` dengan Hook `useMemo([])`.
*   `Frontend/frontend/src/pages/mentor/Qna.tsx`: Membungkus `token` dengan `useMemo([])`.
*   `Frontend/frontend/src/pages/admin/AdminInbox.tsx`: Membungkus `token` dengan `useMemo([])`.

**Latar Belakang Teknis:**
Dengan injeksi `useMemo([])`, variabel token dan profil ditarik dari `localStorage` **hanya satu kali** saat layar dikunjungi (*Mounting*). Perubahan state apapun di masa depan (bahkan jika ada Tab 2 yang mengubah Token) tidak lagi mampu mencemari variabel lokal milik Tab 1! Tab 1 akan mengunci mati memori identitasnya sendiri.

## 4. Manual Testing Plan
1. Buka 2 Tab di browser biasa (bukan Incognito).
2. Login Tab 1 sebagai Mentor. Masuk ke halaman Inbox.
3. Login Tab 2 sebagai User.
4. Pada Tab 2, mulai chat dengan Mentor lain.
5. Cek Tab 1. Ia **tidak akan lagi menarik atau menampilkan** chat milik Tab 2, karena ingatan Tab 1 (`token`) sudah terputus dari interferensi eksternal!
6. Pesan tidak akan lagi "Kesasar" ke Inbox yang salah, dan Header Chat tidak akan lagi menampilkan nama diri sendiri.
