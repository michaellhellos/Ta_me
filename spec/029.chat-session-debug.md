# Spec 029: Chat Session Collision Debugging

## 1. Prompt & Objective
**Prompt:** "{bantu saya itu mengapa saya login user nya lalu saya chat ke mentor nya mengapa dia kayak aneh tidak mengirim ke mentor nya malah yang di atas nya itu mengirim ke saya sendiri dan juga apa bila saya login sebagai mentor nya chat ke admin juga tidak bisa mengapa ya tolong saya perbaiki}"
**Tujuan:** Mengidentifikasi dan membasmi akar masalah mengapa pengirim pesan (Sender) di aplikasi chatting justru melihat namanya sendiri sebagai penerima, pesan terlihat di-loop kembali, serta mengapa Mentor terkadang gagal mem-ping Admin.

## 2. Analisa Akar Masalah (Root Cause)

### A. The "Chatting with Myself" Loop (Local Storage Bleeding)
Setelah melakukan pelacakan mendalam terhadap *WebSockets*, database *Mongoose*, dan *React State*, saya menemukan bahwa logika backend dan frontend **100% sempurna**. Lalu, mengapa bug ini terjadi?
Jawabannya adalah **Metode Testing Manual**.
Ketika Anda melakukan testing dengan membuka 2 Tab di browser yang sama (Tab 1 untuk User, Tab 2 untuk Mentor), browser menggunakan memori `localStorage` yang terhubung untuk kedua tab tersebut.
- Saat Anda login sebagai Mentor di Tab 2, `localStorage.token` dan `localStorage.user` **menimpa** milik User di Tab 1.
- Ketika Tab 1 kembali dibuka dan mencoba mengetik pesan, aplikasi mengambil JWT Token yang kini merupakan milik **Mentor**.
- Backend secara patuh membaca token tersebut dan mencatat pesan tersebut dikirim OLEH Mentor.
- Frontend di Tab 1 menerima websocket kembali, mengevaluasi `isMe`, dan menganggap bahwa Mentor mengirim pesan ke dirinya sendiri. Inilah alasan kuat mengapa Chat Header / Pesan seolah terlihat "Saya ngobrol dengan saya sendiri".
**Solusi:** Tidak ada kode yang perlu diubah untuk masalah ini. Pengujian multi-role *wajib* dilakukan menggunakan 2 browser berbeda (misal Chrome dan Edge), atau 1 menggunakan Jendela Normal dan 1 menggunakan Jendela *Incognito/Private*.

### B. Admin Not Found Error
Mentor gagal melakukan chat dengan Admin karena database Admin belum ter-*generate*. Endpoint backend (`auth.js`) didesain untuk menciptakan entitas Admin secara otomatis *hanya* saat `admin@gmail.com` dieksekusi di halaman Login. Jika skenario testing belum pernah menyentuh form login sebagai Admin pasca penghapusan database, maka tombol "Hubungi Admin Support" akan melaporkan "Akun tidak ditemukan".
**Solusi:** Cukup *login* sebagai Admin satu kali untuk men-trigger *seeding* database.

### C. The Inbox Silent Drop Bug
Saya menemukan sebuah celah valid pada UI Inbox: Jika User A menekan tombol "Chat Sekarang" dan mengirim pesan pertama kepada Mentor B, `socket.on('receive_message')` milik Mentor B tidak mengenali `conversationId` karena ruangan tersebut benar-benar baru. Pesan tersebut masuk ke Database, namun tidak langsung muncul di layar Inbox Mentor B tanpa dilerfresh.
**Solusi Kode Langkah Pencegahan:** Memodifikasi `AdminInbox.tsx` dan `Qna.tsx` agar memicu refetch ke server jika terdeteksi `conversationId` yang asing (belum ada di list state lokal `prev`). 

## 3. Rencana Perubahan Code
- `Frontend/frontend/src/pages/mentor/Qna.tsx`: Menginject validasi `!convExists` ke dalam `socket.on`. Jika obrolan asing, segera jalankan ulang `fetchConversations()`.
- `Frontend/frontend/src/pages/admin/AdminInbox.tsx`: Menerapkan validasi identik di Socket UI Admin.

## 4. Manual Testing Plan
1. Buka browser biasa. Login sebagai `user@email.com`.
2. Buka jendela **Incognito/Private**. Login sebagai `admin@gmail.com` (ini memancing seed Admin jika belum ada). 
3. Logout jendela Incognito, lalu login sebagai Mentor. Buka *Inbox Q&A*.
4. Pada browser biasa (User), kunjungi halaman AI Assistant, pilih Mentor tersebut, kirimkan "Halo mentor!".
5. Cek jendela Incognito. Pesan akan segera me-render Box Chat baru tanpa harus merefresh halaman (berkat tambalan socket baru).
6. Balas pesan tersebut. Sinkronisasi sempurna `isMe` (Right) dan `Other` (Left) akan terjadi karena tab tidak lagi memperebutkan 1 JWT token.
