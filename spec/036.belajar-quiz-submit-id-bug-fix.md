## 2. Tujuan yang Dipahami

**Memperbaiki bug gagal submit quiz** di halaman `Belajar.tsx` yang mengeluarkan error: *"userId dan materiId wajib diisi"*.

User kebingungan karena mereka sudah login (sehingga seharusnya ada `userId`), dan mereka sedang mengerjakan soal tertentu (sehingga seharusnya ada `id soal` / `materiId`). 

**Goal utama:** Memastikan ID user dan ID materi terkirim dengan benar ke backend tanpa error.

---

## 3. Codebase Analysis (Kenapa Bug Ini Terjadi)

Setelah menelusuri alur data dari login hingga submit quiz, ditemukan akar masalahnya:

### 3.1 Masalah pada `userId` (Akar Utama)

1. **Backend Login (`auth.js` line 100-108):**
   Saat user login, backend mem-format response user dan mengubah `_id` menjadi `id`:
   ```javascript
   user: {
     id: user._id,       // <-- Dikirim sebagai 'id'
     name: user.name,
     email: user.email,
     role: user.role,
     balance: user.balance
   }
   ```
2. **Frontend Login (`Login.tsx` line 43):**
   Response ini disimpan persis apa adanya ke dalam `localStorage` dengan key `"user"`.
3. **Frontend Submit Quiz (`Belajar.tsx` line 216-217):**
   Saat submit quiz, kode mencoba mengambil `_id`:
   ```typescript
   const parsedUser = JSON.parse(userData);
   const userId = parsedUser._id;  // <-- UNDEFINED!
   ```
   Karena object-nya hanya punya `id`, maka `parsedUser._id` menghasilkan `undefined`.

### 3.2 Kenapa Error-nya Menyebutkan `materiId` Juga?

Di backend (`nilaiController.js` line 8-13):
```javascript
if (!userId || !materiId) {
  return res.status(400).json({
    success: false,
    message: "userId dan materiId wajib diisi",
  });
}
```
Meskipun `materiId` terkirim dengan benar (dari `selectedMateri._id`), karena `userId` adalah `undefined`, kondisi `IF` tersebut terpenuhi (TRUE). Backend menggunakan pesan error gabungan *"userId dan materiId"*, sehingga user mengira keduanya rusak. Faktanya, hanya `userId` yang kosong.

### 3.3 Tentang `id soal` (`materiId`)

Di `Belajar.tsx` (line 226):
```typescript
materiId: selectedMateri._id,
```
Data `selectedMateri` berasa dari `GET /api/materi/materi` yang mana adalah data langsung dari MongoDB. MongoDB secara bawaan selalu menyertakan `_id`. Jadi kode `selectedMateri._id` sudah benar dan tidak menyebabkan error (kecuali format ID-nya dianggap tidak valid oleh mongoose, tapi dalam kasus ini karena undefined `userId` yang kena duluan).

## 4. Perubahan yang Akan Diimplementasi

### 4.1 Rangkuman

Cukup mengubah cara frontend mengambil `userId` dari `localStorage` agar lebih aman dengan menggunakan *fallback*.

| # | File | Perubahan | Why |
|---|------|-----------|-----|
| A | `Belajar.tsx` | Ubah letak ekstraksi `userId` menjadi `parsedUser._id || parsedUser.id` | Menjamin `userId` terambil apapun key-nya (baik itu `id` dari response login, atau `_id` jika direstrukturisasi di masa depan) |

**Total: 0 new, 1 modified (frontend), 0 deleted**

### 4.2 Detail Perubahan

#### [MODIFY] `Frontend/frontend/src/pages/Belajar.tsx`

**Sebelum:**
```typescript
    const parsedUser = JSON.parse(userData);
    const userId = parsedUser._id;
```

**Sesudah:**
```typescript
    const parsedUser = JSON.parse(userData);
    // Fallback: dukungan 'id' dari /api/auth/login dan '_id' dari standard Mongo
    const userId = parsedUser._id || parsedUser.id; 
    
    if (!userId) {
      setToast({ message: "Sesi tidak valid, harap login ulang", type: "error" });
      return;
    }
```

## 5. Logic / Pseudocode

**Flow Submit Quiz:**
1. Jawab soal → Klik Submit
2. Ambil string `"user"` dari localStorage
3. `JSON.parse` menjadi objek `parsedUser`
4. Cari ID user: `parsedUser._id` (jika tidak ada) $\rightarrow$ cari `parsedUser.id`
5. Jika hasil pencarian gagal (`undefined`), batalkan pengiriman dan minta user login.
6. Buat payload `{ userId: '...', materiId: selectedMateri._id, score: ..., totalSoal: ... }`
7. Kirim ke backend. Backend memvalidasi `userId` dan `materiId`. Payload valid $\rightarrow$ Sukses.
