## 1. Prompt / Task

```
Buatkan forum supaya bisa upload gambar, bisa upload file, dan juga ENSURE SECURITY 
dari frontend dan backend. Jadi bisa upload file & emoji, dan juga gunakan WYSIWYG 
(yang gratis dan tidak perlu API KEY).
```

---

## 2. Tujuan yang Dipahami

Mengubah **forum diskusi** (saat ini masih placeholder "akan segera hadir") menjadi forum yang **fungsional** dengan kemampuan:

| # | Capability | Keterangan |
|---|-----------|------------|
| 1 | **WYSIWYG Editor** | Rich-text editor gratis (tanpa API key) menggantikan `<textarea>` biasa |
| 2 | **Image Upload** | User/Mentor bisa upload gambar langsung dari editor atau tombol |
| 3 | **File Upload** | User/Mentor bisa attach file (PDF, DOC, dll) ke post |
| 4 | **Emoji Support** | Emoji picker terintegrasi di editor |
| 5 | **Security (FE)** | Validasi MIME type, ukuran file, sanitasi HTML output |
| 6 | **Security (BE)** | Validasi file type, max size, sanitasi HTML, rate limiting |

---

## 3. Kenapa Tujuan Ini?

1. **Forum saat ini tidak berfungsi** — Tab "Forum Diskusi" di halaman Komunitas (`Komunitas.tsx`) hanya menampilkan placeholder text.
2. **Mentor create post terbatas** — `Createforum.tsx` hanya mengirim plain text tanpa kemampuan upload.
3. **Model sudah disiapkan** — `CommunityPost.js` sudah punya field `image` dan `file` (tipe String/URL), tapi belum ada mekanisme upload yang sebenarnya. Saat ini field ini selalu `null`.
4. **Keamanan belum ada** — Tidak ada validasi file, tidak ada sanitasi HTML, tidak ada proteksi terhadap file berbahaya.
5. **UX basic** — Textarea biasa tidak cukup untuk forum diskusi yang engaging. WYSIWYG dan emoji meningkatkan kualitas konten.

#### 5.2.3 BACKEND — [NEW] `middleware/sanitize.js`

**Kenapa:** Sanitasi HTML dari WYSIWYG editor sebelum disimpan ke database. Mencegah XSS dan script injection.

```
Pseudocode:
─────────────────────────────────────────────────
1. Import sanitize-html

2. Define sanitizeConfig = {
     allowedTags: ['p', 'br', 'strong', 'em', 'u', 's', 'h1', 'h2', 'h3',
                   'ul', 'ol', 'li', 'blockquote', 'a', 'img', 'pre', 'code'],
     allowedAttributes: {
       'a': ['href', 'target', 'rel'],
       'img': ['src', 'alt', 'width', 'height']
     },
     allowedSchemes: ['http', 'https'],
     // BLOCK: script, iframe, style, onXXX attributes
   }

3. Export function sanitizeContent(html):
     IF html is empty → return ''
     return sanitizeHtml(html, sanitizeConfig)
─────────────────────────────────────────────────
```

─────────────────────────────────────────────────
1. Middleware: auth → mentorOnly (atau auth saja jika semua boleh post)
2. Receive body: { content (HTML), attachments (array of urls) }

   - content tidak boleh kosong setelah strip HTML tags
   - attachments max 5 items
   - Setiap attachment URL harus valid (starts with /uploads/)
4. SANITIZE:
   - content = sanitizeContent(content)
5. CREATE post:
   - author: req.user._id
   - authorName: req.user.name
   - role: req.user.role
   - content: sanitized HTML
   - attachments: validated attachments array
6. Return created post
─────────────────────────────────────────────────

Pseudocode — DELETE /api/community/upload/:filename (NEW):
─────────────────────────────────────────────────
1. Middleware: auth
2. Validate filename (no path traversal: ../ or absolute paths)
3. Check file exists in uploads/
4. Delete file from disk
5. Return success

```

## 6. Security Details

### 6.1 Backend Security Checklist

| # | Check | Implementation |
|---|-------|---------------|
| 1 | **File Type Whitelist** | `multer` fileFilter hanya izinkan MIME types yang di-whitelist |
| 2 | **Max File Size** | Images: 5MB, Files: 10MB via `multer` limits |
| 3 | **Filename Sanitization** | Rename ke UUID + extension, hapus original filename dari disk path |
| 4 | **Path Traversal Prevention** | Validate filename di delete endpoint (no `../` atau absolute paths) |
| 5 | **HTML Sanitization** | `sanitize-html` strip semua tag berbahaya (`<script>`, `<iframe>`, event handlers `onXXX`) |
| 6 | **XSS Prevention** | Hanya allow safe HTML tags dari WYSIWYG output |
| 7 | **Auth Required** | Semua upload/create endpoint require JWT auth middleware |
| 8 | **Role Check** | Create post dan upload restricted ke authorized roles |
| 9 | **Content Validation** | Empty content rejection, max content length check |
| 10 | **Static File Isolation** | Upload folder served as static, terpisah dari app code |

### 6.2 Frontend Security Checklist

| # | Check | Implementation |
|---|-------|---------------|
| 1 | **Pre-upload MIME Validation** | Check `file.type` sebelum upload ke server |
| 2 | **Pre-upload Size Validation** | Check `file.size` sebelum upload ke server |
| 3 | **Safe HTML Rendering** | Content dari server dianggap sudah di-sanitize, tapi tetap hanya render di `dangerouslySetInnerHTML` (React auto-escapes di tempat lain) |
| 4 | **Token Required** | Semua API calls include `Authorization: Bearer {token}` |
| 5 | **Input Validation** | Disable submit jika konten kosong |

### 6.3 Kenapa Security Approach Ini?

**Double Validation (FE + BE):**
- Frontend validation = UX responsiveness (cepat feedback ke user)
- Backend validation = security layer sesungguhnya (FE bisa di-bypass oleh attacker)

**HTML Sanitization di Backend:**
- WYSIWYG output HTML bisa di-inject oleh user (inspect element, Postman)
- Server HARUS sanitize sebelum save ke DB
- Frontend render via `dangerouslySetInnerHTML` relatif aman JIKA data dari DB sudah sanitized

**UUID Filename:**
- Mencegah filename collision
- Mencegah path traversal via crafted filename
- Tidak expose informasi dari original filename

## 9. Tracing — Request Flow

### Trace 1: Mentor Uploads Image and Creates Post

```
[1] Mentor clicks 📷 button
    └─ <input type="file" accept="image/*"> triggers

[2] File selected: "photo.jpg" (2MB, image/jpeg)
    └─ Frontend validate:
       ├─ file.type = "image/jpeg" ✅ (in allowed list)
       └─ file.size = 2097152 ✅ (< 5242880)

[3] FormData created: { image: File }
    └─ POST http://localhost:5000/api/community/upload
       Header: Authorization: Bearer eyJhbG...

[4] Backend receives request
    ├─ auth middleware → verify JWT → load User from DB → req.user = { _id, name, role }
    ├─ multer middleware:
    │   ├─ fileFilter: "image/jpeg" ✅ in whitelist
    │   ├─ limits: 2MB < 5MB ✅
    │   └─ storage: save as "a1b2c3d4.jpg" in uploads/
    └─ Route handler:
        └─ Response: { success: true, url: "/uploads/a1b2c3d4.jpg", originalName: "photo.jpg", ... }

[5] Frontend receives URL
    ├─ Quill editor: insert <img src="http://localhost:5000/uploads/a1b2c3d4.jpg">
    └─ attachments state: [..., { url: "/uploads/a1b2c3d4.jpg", originalName: "photo.jpg", ... }]

[6] Mentor clicks "BROADCAST SEKARANG"
    └─ POST http://localhost:5000/api/community/create
       Body: {
         content: "<p>Sinyal BTC hari ini:</p><img src=\"/uploads/a1b2c3d4.jpg\"><p>Bullish!</p>",
         attachments: [{ url: "/uploads/a1b2c3d4.jpg", originalName: "photo.jpg", type: "image", ... }]
       }

[7] Backend receives request
    ├─ auth middleware ✅
    ├─ mentorOnly middleware ✅ (role = "mentor")
    └─ Route handler:
        ├─ Validate: content not empty (strip tags → "Sinyal BTC hari ini:Bullish!") ✅
        ├─ Sanitize HTML: <p>, <img> → ALLOWED ✅, no <script> ✅
        ├─ Validate attachments: 1 item, URL starts with "/uploads/" ✅
        ├─ Create CommunityPost in MongoDB
        └─ Response: { message: "Broadcast berhasil dibuat", post: {...} }
```

```

## 11. Open Questions (Require Clarification)

> [!IMPORTANT]
> Pertanyaan berikut perlu dijawab sebelum implementasi.

1. **Siapa yang boleh buat post di forum?**
   - Saat ini `POST /create` pakai `mentorOnly` middleware → hanya mentor & admin.
   - Apakah user biasa juga boleh post di forum? Atau forum hanya untuk broadcast dari mentor?

2. **Apakah perlu fitur komentar di post?**
   - Model sudah ada `commentsCount`, tapi belum ada Comment model atau route.
   - Apakah komentar juga perlu upload/WYSIWYG? Atau plain text saja?

3. **Storage approach:**
   - Saat ini rencana simpan file di local disk (`/uploads/`).
   - Ini cocok untuk development. Untuk production, apakah perlu cloud storage (S3, Cloudinary)?
   - Atau local disk cukup untuk sekarang?

4. **Max attachments per post?**
   - Proposal: max 5 attachments (image + file combined) per post.
   - Apakah ini cukup? Atau perlu lebih/kurang?

5. **Backward compatibility:**
   - Existing posts di DB punya `image` dan `file` field (selalu null saat ini).
   - Apakah ada data lama yang perlu dimigrasikan, atau semua post existing bisa diabaikan?
