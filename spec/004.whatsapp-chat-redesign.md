## 1. Prompt / Task

```
Bantu saya untuk chat nya antara mentor dan user nya, chat nya itu kan sudah 
real time tetapi bantu saya buatkan dia seperti di WhatsApp untuk chat antara 
mentor dan user nya dan terdapat centang centang nya begitu.

Bantu saya apabila saya kirim chat (role saya user) ke mentor nya:
• Apabila mentor nya BELUM membalas chat saya dan BELUM membaca nya → centang 1 berwarna abu
• Apabila TELAH dibaca → centang 2 berwarna biru

Chat nya bisa kiri kanan:
• Apabila SAYA chat mentor → tampilan nya di sebelah KANAN
• Apabila mentor nya menjawab → tampilan nya di sebelah KIRI
• Di halaman mentor nya: chat mentor di sebelah KANAN, user read di sebelah KIRI

Logika simple: apabila SAYA kirim pesan → tampil di KANAN, apabila SAYA terima → tampil di KIRI.
```

## 3. Kenapa Tujuan Ini?

### 3.1 Read Receipt TIDAK Persist ke Database

**BUG KRITIS #1:** Saat ini, `message_read` socket event di `server.js` (line 89-91):

```javascript
socket.on("message_read", (data) => {
  io.to(data.conversationId).emit("message_read_update", data);
});
```

Event ini **hanya broadcast ke socket room** tapi **TIDAK update `isRead` di MongoDB**. Artinya:
- ✅ Real-time: centang biru muncul saat kedua user online
- ❌ Persist: refresh halaman → semua centang kembali ke abu (single ✔)
- ❌ History: buka chat lagi → semua pesan tampil belum dibaca

### 3.2 Duplicate Message pada Sender

**BUG KRITIS #2:** Di `server.js` line 84-86:

```javascript
socket.on("send_message", (data) => {
  io.to(data.conversationId).emit("receive_message", data);
  //  ↑ io.to() broadcasts ke SEMUA socket di room, TERMASUK sender
});
```

Dan di `Chat.tsx` line 83-88:

```tsx
socket.emit("send_message", { ...newMessage, conversationId });
setMessages((prev) => [...prev, newMessage]);
// ↑ Sender menambahkan pesan secara lokal
```

Sender **menambahkan pesan dua kali**: sekali dari `setMessages` lokal, sekali dari `receive_message` socket event. Hasilnya: **pesan muncul ganda di layar sender.**

### 3.3 Type Mismatch: Frontend `status` vs Backend `isRead`

Frontend `Message` type punya field `status?: "sent" | "read"`, tapi backend `Message.js` punya `isRead: Boolean`. Backend populate response mengirim `isRead` (true/false), bukan `status`. Frontend convert ini di client side tapi tidak konsisten — `receive_message` via socket mungkin tidak punya `isRead`, hanya punya format dari `Message.create()` response.

### 3.4 Chat Page Tidak Informatif

- Header hanya menampilkan "Chat Room" → tidak tahu sedang chat dengan siapa
- Tidak ada timestamp pada message bubbles
- Tidak ada date separator
- Tidak auto-scroll ke pesan terbaru
- Tidak ada typing indicator

| Event | Direction | Payload | Masalah |
|-------|-----------|---------|---------|
| `join_conversation` | C→S | `conversationId` | ✅ OK |

| `message_read` | C→S | `{ conversationId, messageId }` | ❌ Tidak persist ke DB |
| `message_read_update` | S→C | `{ conversationId, messageId }` | ⚠️ Works real-time tapi hilang saat refresh |

### 4.5 Message Model

```javascript
{
  conversationId: ObjectId → Conversation (required)
  sender:         ObjectId → User (required)
  receiver:       ObjectId → User (required)
  text:           String (required)
  isRead:         Boolean (default: false)       // ← KEY FIELD
  createdAt:      Date (auto, via timestamps)    // ← sudah ada, tidak dipakai frontend
  updatedAt:      Date (auto, via timestamps)
}
```

### 4.6 Current Read Receipt Flow — Broken

```
CURRENT:
1. User A sends → Message saved with isRead: false
2. User B opens chat → clicks msg → socket.emit("message_read", { messageId })
3. Server broadcasts "message_read_update" to room
4. User A sees ✔✔ (real-time via socket state)
5. ❌ PROBLEM: isRead STILL false in MongoDB
6. ❌ User A refreshes → ✔✔ gone, back to ✔
```

### 4.7 Frontend Type vs Backend Field

```
FRONTEND (Chat.tsx line 10-18):
  type Message = {
    _id?: string;
    text: string;
    sender: { _id: string; name: string };  // ← missing role
    status?: "sent" | "read";               // ← frontend-only field
    // ❌ MISSING: isRead, createdAt
  };

BACKEND sends (from GET /message/:id):
  {
    _id, conversationId, sender: { _id, name, role },
    receiver, text, isRead, createdAt, updatedAt
  }
```

// 5️⃣ MARK ALL MESSAGES AS READ (saat buka chat)
router.put("/mark-read/:conversationId", auth, async (req, res) => {
  try {
    const userId = req.user.id;
    
    // Update semua pesan di conversation ini yang:
    // - receiver adalah user yang sedang login (bukan sender)

    });

### Trace 2: User Kirim Pesan → Centang Abu → Mentor Baca → Centang Biru

```
[1] User types "Mau tanya soal BTC" → presses Enter
    └─ sendMessage()
    └─ POST /api/chat/message { conversationId: "conv456", text: "Mau tanya soal BTC" }
    └─ Backend: Message.create({ 
         sender: "user789", receiver: "mentor123", 
         text: "Mau tanya soal BTC", isRead: false   ← DEFAULT
       })
    └─ Response: { data: { _id: "m3", ..., isRead: false } }

[2] Frontend (sender):
    ├─ setMessages(prev => [...prev, m3])
    ├─ m3 appears di KANAN (isMe = true)
    ├─ m3.isRead = false → ✔ ABU-ABU (belum dibaca)
    └─ socket.emit("send_message", { ...m3, conversationId })

[3] Server.js:
    └─ socket.to("conv456").emit("receive_message", m3)
    └─ ✅ Only MENTOR receives (not sender, karena socket.to)

[4] Mentor's Chat.tsx receives "receive_message":
    ├─ setMessages(prev => [...prev, m3])
    ├─ m3 appears di KIRI (isMe = false)
    └─ Auto-mark read: socket.emit("message_read", { messageId: "m3" })

[5] Server.js → message_read:
    ├─ await Message.findByIdAndUpdate("m3", { isRead: true })  ← PERSISTED ✅
    └─ io.to("conv456").emit("message_read_update", { messageId: "m3" })

[6] User's Chat.tsx receives "message_read_update":
    └─ messages.map(msg => msg._id === "m3" ? {...msg, isRead: true} : msg)
    └─ m3 check: ✔ abu → ✔✔ BIRU 🔵

[7] RENDER updated (User side):
    ┌─────────────────────────────────────────────┐
    │                   ┌──────────────────────────┐
    │                   │ Mau tanya soal BTC       │ ← KANAN
    │                   │              14:30 ✔✔ 🔵 │ ← check BIRU
    │                   └──────────────────────────┘
    └─────────────────────────────────────────────┘

[8] RENDER (Mentor side):
    ┌─────────────────────────────────────────────┐
    │  ┌──────────────────────────────┐           │
    │  │ Mau tanya soal BTC    14:30 │           │ ← KIRI (dari user)
    │  └──────────────────────────────┘           │
    └─────────────────────────────────────────────┘
```
